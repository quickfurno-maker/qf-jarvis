# ADR-0037 — Stage 3.4.4: the internal projection runner

**Status:** Accepted (2026-07-20, under the owner's standing Stage 3.4 authorization)
**Deciders:** Owner
**Phase:** 3 — Stage 3.4.4 (the internal projection runner; the worker/scheduler that drives it is later)

**Relates to:** [ADR-0019](./ADR-0019-durable-event-store-and-persistence.md) (event store) · [ADR-0021](./ADR-0021-processing-retries-dead-letters-and-replay.md) (processing, retries, dead letters, replay) · [ADR-0022](./ADR-0022-projections-ordering-and-rebuild-determinism.md) (projection ordering, rebuild determinism) · [ADR-0034](./ADR-0034-stage-3-4-projections-checkpoints-and-bounded-retries.md) (Stage 3.4.1 checkpoints, the transaction protocol, the retry bound) · [ADR-0035](./ADR-0035-stage-3-4-2-projection-registry.md) (Stage 3.4.2 registry) · [ADR-0036](./ADR-0036-stage-3-4-3-gap-free-projection-ordering.md) (Stage 3.4.3 gap-free projection positions)

## Context

The Stage 3.4 foundation is now in place and merged: the checkpoint/attempt schema and repositories (3.4.1, ADR-0034, migration `0004`), the internal immutable projection registry (3.4.2, ADR-0035), and the gap-free, commit-ordered projection **position** cursor with its metadata reader (3.4.3, ADR-0036, migration `0005`). ADR-0034 §5 fixed the **transaction protocol** the runner must follow (one outer transaction, a `SAVEPOINT` around the handler, bookkeeping before `COMMIT`), §4 fixed the **lock mechanism** (`pg_try_advisory_xact_lock`, keyed on name+version only), §6 fixed the **retry bound** at five, and §7 explicitly **deferred the exact backoff-interval constants to Stage 3.4.4**. What remained un-built was the runner itself: the code that resolves a projection, takes the lock, reads exactly one position, invokes the handler, and records the outcome durably and atomically.

This slice implements that runner **as internal vocabulary only** — no worker loop, no scheduler, no HTTP, no new migration, and no package-root export. It is the smallest slice that makes ADR-0034 §5's protocol executable against real PostgreSQL, and it locks the parameters ADR-0034 left open.

## Decision

### 1. One invocation processes exactly one position, in one transaction

`runProjectionOnce({ pool, registry, name, version, now })` accepts the projection identity and an **injected canonical `now`** (never a wall clock), resolves the definition from the registry, and **requires the requested version to match**; an unknown name or a version mismatch fails closed with the fixed code `projection-unknown-definition`. It then acquires **one** pool client and runs the ADR-0034 §5 protocol on it directly — it does **not** use the shared `withTransaction`/`withClient` helpers, because `withTransaction` would roll back the deterministic failure bookkeeping that must survive, and `withClient`'s `finally` release could mask the primary error and cannot express destructive disposal. The transaction is explicit:

1. `BEGIN`; `SET TRANSACTION ISOLATION LEVEL READ COMMITTED` (the helpers set no isolation, so the runner sets it itself);
2. derive and attempt the transaction-scoped advisory lock; if it is unavailable, return **`busy`** with **no mutation**;
3. lazily create the checkpoint (`INSERT ... ON CONFLICT DO NOTHING`) then read it `FOR UPDATE`;
4. **reconcile** the checkpoint against its attempt rows **before** returning any `blocked`, `retry-pending`, or writing outcome (see §5);
5. compute the read position as `last_position + 1` and resolve it with the ADR-0036 metadata reader (`readEventAtPosition`) — **never** the raw `event.sequence`; a run is **caught-up** only when there is no next position **and** no contradictory failed/blocked state;
6. `SAVEPOINT projection_handler`; invoke the registered handler with the **same** client.

The result is returned **only after a confirmed `COMMIT`** (or, for a read-only outcome, after the outer transaction is confirmed closed). The runner never manufactures a durable-looking result it did not durably observe.

### 2. Advisory-lock key — deterministic, name+version, int8-exact

The lock argument (`projection-lock-key.ts`) is `asIntN(64, be_uint64(SHA-256("qf-jarvis/projection-lock/v1:" + name + ":" + version)[0..7]))`, bound to `pg_try_advisory_xact_lock($1::bigint)` as its **decimal string** — it never passes through a JavaScript `number`, which cannot hold the full int8 range. It is domain/version-prefixed and derived from the validated name **and** version (a version bump is a distinct lock), and it is free of any event/payload/subject/caller data. Fixed vectors: `event-type-activity:1 → -5669050953429467864`, `daily-event-acceptance:1 → 1246584766324522588`, `event-type-activity:2 → -9050093009359467535`. Domain separation prevents **namespace reuse** — this derivation will not accidentally land in another subsystem's advisory-lock convention — but it **cannot mathematically guarantee** the absence of a 64-bit collision with every advisory-lock user everywhere. Any residual collision is nonetheless **liveness-only**: two colliding pairs would only make one projection observe `busy` while the other holds the lock (it simply runs next tick). Correctness is guarded by each projection's own checkpoint `FOR UPDATE` row lock, not by the advisory key.

### 3. Backoff — deterministic equal-jitter, the constants ADR-0034 §7 deferred

The `next_attempt_at` interval (`projection-backoff.ts`) is deterministic equal-jitter with `BASE_MS = 1000`, `MULTIPLIER = 8`, `MAX_MS = 300000` (five minutes, honouring ADR-0021 §2's cap). For failure `n`: `upper = min(MAX, BASE × 8^(n-1))`, `lower = ceil(upper / 2)`, and `delay = lower + (uint32(SHA-256("qf-jarvis/projection-backoff/v1:" + name + ":" + version + ":" + position + ":" + failure)[0..3]) mod (upper − lower + 1))`. It uses **no** `Math.random`, no RNG, no mutable clock, and no caller text; the jitter is a pure function of `(name, version, position, failureNumber)`, so it is identical on every process and every restart — a durable schedule cannot be reset by a crash-loop. Failures **1–4 schedule** a retry; **failure 5 blocks** (and schedules nothing). It **fails closed** (throws `ProjectionInputError`) if `now + delay` is not a representable canonical instant. Fixed vectors (`event-type-activity`, v1, position 7): failure 1 → 551 ms, 2 → 4423 ms, 3 → 54533 ms, 4 → 166001 ms.

### 4. The result is a frozen, primitives-only union

`runProjectionOnce` returns one of exactly seven frozen variants (`projection-run-result.ts`): `busy`, `caught-up`, `succeeded`, `retry-scheduled`, `blocked-now`, `blocked-existing`, `retry-pending`. Every variant carries `projectionName` and `projectionVersion`. The union contains **only** primitives — `bigint` positions, canonical-instant strings, validated counts and the closed `safe_error_code` set. It carries **no** `Date`, `Error`, `cause`, stack, `SQLSTATE`, driver message, payload, or caller text, and each result object is `Object.freeze`d. The caller learns the outcome and the durable facts, and nothing a hostile handler or driver could smuggle through.

### 5. Reconciliation — a stored count with no matching attempts is divergence, not progress

Before returning any `blocked`, `retry-pending`, or writing outcome, the runner **reconciles** the checkpoint against `qf_jarvis.projection_attempt`. For an **active** checkpoint with failed count `k`, the pending position is `last_position + 1`, and there must be **exactly `k`** attempt rows numbered `1..k`, **all `failed`** — a gap, a duplicate, a `succeeded` row, or a future/missing position is a divergence. For a **blocked** checkpoint, there must be **exactly five** `failed` rows `1..5` and the blocked position must resolve to a real event. Any mismatch rolls back the **whole** transaction, throws a fresh `projection-attempt-checkpoint-divergent`, mutates nothing, and retains no partial value. **A retry count greater than zero with no corresponding event position is divergence, not caught-up** — the runner never silently treats an inconsistent checkpoint as progress.

### 6. Error classification — the caller's object is never preserved, and it never dictates a classification

A thrown handler or driver value is **always discarded** — never re-thrown, never wrapped as `cause`, never copied by message, stack, `cause`, or enumerable field, even if it forges an `instanceof` subclass. Classification is by **phase**, not by the object's identity:

Classification is a closed **tri-state** inspection of the caught value's `code` (`inspectSqlstate`), because an **absent** SQLSTATE is materially different from an **unreadable** one:

- **Absent** — the value has **no own `code` property at all** (an ordinary `Error`, a thrown primitive, a plain object). This is a **single deterministic-handler-failure** bucket: `ROLLBACK TO SAVEPOINT projection_handler`, append **one** attempt with the fixed code `projection-handler-failed`, then schedule (failures 1–4) or block (failure 5), then `COMMIT`.
- **Valid** — an own **data-property** `code` that is a well-formed five-character SQLSTATE (`^[0-9A-Z]{5}$`). The deterministic/infrastructure tables then decide: a recognised handler-side code (`23505`, `23514`, `23503`, `23502`, `22*`, `42501`, `42P01`, `42703`, `42883`) is the deterministic bucket above; a recognised infrastructure code (`08*`, `40P01`, `40001`, `53*`, `55P03`, `57014`, `57P0*`, `58*`, `XX*`) **or any well-formed but unrecognised** code is **conservative infrastructure**.
- **Invalid / unreadable** — an own `code` that is an **accessor (getter/setter)**, a **non-string**, a **malformed/empty** string, or a value that could not even be inspected. This is **conservative infrastructure**. An accessor is explicitly **not** treated as "no code": a hostile object cannot downgrade itself to a deterministic classification by hiding its `code` behind a getter.

Inspection eligibility is **property-bearing** — a non-null **object OR a function** — not merely `typeof === 'object'`. A thrown **function** is a JavaScript value that can carry an own data/accessor `code` (and can itself be a revoked `Proxy` or a hostile descriptor trap), so it is inspected exactly like an object; only a **true primitive** is `absent`. The function is never invoked.

An **infrastructure** classification records **no** attempt: the outer transaction is rolled back and a fixed-code `projection-infrastructure-failed` is thrown. The SQLSTATE is read **only** from an own **data** descriptor — the getter of an accessor `code` is **never invoked**, and `valueOf`/`toString` are never called. `Object.getOwnPropertyDescriptor` is itself **wrapped in `try/catch`**, so a **revoked object or function `Proxy`**, or a hostile `getOwnPropertyDescriptor` trap, cannot leak its thrown value: the inspection fails closed to `invalid` (infrastructure), and no raw `TypeError` or `Proxy`-thrown value ever escapes the runner. A runner/bookkeeping/invariant failure (as opposed to a handler failure) always rolls back the outer transaction and **never** consumes or persists a handler attempt; the runner always executes the `ROLLBACK TO SAVEPOINT` after a handler failure **before** deciding bookkeeping, and if that rollback itself fails it is treated as infrastructure.

### 7. Unknown COMMIT and destructive client disposal

The runner tracks the outer transaction's lifecycle **explicitly** — `not-started` before `BEGIN`, `open` once `BEGIN` succeeds, `closed` after a confirmed `COMMIT` or confirmed outer `ROLLBACK`, and `unknown` after any failed or ambiguous `BEGIN`/`COMMIT`/`ROLLBACK` — so release-vs-destroy is **proven, not assumed**. A **normal** `client.release()` happens **only** when the transaction is `closed` (or never started). In every other case — a `COMMIT` that throws or is ambiguous, an uncertain `BEGIN`, a failed outer `ROLLBACK`, or an otherwise `open`/`unknown` state — the client is **destroyed** via `client.release(true)` so a poisoned or transaction-bearing connection is never returned to the pool.

A **final safety boundary** wraps the whole run: if any **unexpected** value escapes while the transaction is still `open` (a latent defect rather than an intentional outcome), the runner attempts an outer `ROLLBACK` (marking `closed` on success, `unknown` on failure), then throws a **fresh** fixed-code `projection-infrastructure-failed`. The unexpected caught value is **never preserved**, and the client is never returned normally merely because some path forgot to record that it must be destroyed. An intentional `ProjectionRunnerError` passes through the boundary **unchanged**, so `projection-commit-outcome-unknown` and `projection-attempt-checkpoint-divergent` are never rewritten. That recognition is **guarded** (`isProjectionRunnerErrorSafely`): a bare `value instanceof ProjectionRunnerError` performs `[[GetPrototypeOf]]` on the caught value, which **throws for a revoked `Proxy`** (object or function), so the check is wrapped in `try/catch` and inspects **only** the prototype relationship — never the caught value's `message`, `stack`, `cause`, `code`, or enumerable fields; anything not safely identified is treated as unexpected and normalised. Each frozen result is built **before** its `COMMIT`, so once a `COMMIT` is confirmed the only remaining step is returning the already-built value.

This boundary is proven against real PostgreSQL by injecting an **unexpected failure at the pre-`COMMIT` result construction** — the `Object.freeze` inside `succeededResult(...)`, which runs **outside every local `executeRun` `try/catch`** and before `COMMIT`. The genuine boundary (not the local repository-phase catch) then rolls back and, when the fallback `ROLLBACK` succeeds, releases the client **normally** (reusable); when the fallback `ROLLBACK` fails, the transaction becomes `unknown` and the client is **destroyed** via `release(true)`. In both cases the caller receives a fresh `projection-infrastructure-failed` and the injected marker never appears in the error's message/code/stack/cause/JSON. A local caught query failure (e.g. on the checkpoint query) is a distinct **repository-phase** containment path and is labelled as such — it does **not** prove the outer boundary.

If the `COMMIT` acknowledgement is lost (the server may or may not have committed), the runner throws a fresh `projection-commit-outcome-unknown`, performs **no** post-commit `ROLLBACK`, destroys the client, and carries **no** caught message, cause, or SQLSTATE; re-invocation is required to observe the durable state, and the checkpoint `FOR UPDATE` + `= last_position + 1` protocol guarantees a re-invocation **cannot double-apply**. Cleanup errors never replace the primary error; the client is released or destroyed **exactly once**; a cleanup failure is swallowed. A `pool.connect()` that fails before any client is obtained surfaces as `projection-runner-unavailable`.

**No database-client-abstraction change was required.** The destructive `client.release(true)` and `client.release()` already typecheck directly against the existing `DatabaseClient` (`pg.PoolClient`) surface; the pool client's destructive release capability is already exposed. This is reported here explicitly because the slice was authorized to make a smallest-possible type-only edit to that abstraction **only if required** — it was not.

### 8. Closed runner-error vocabulary; internal only

The runner throws **only** five closed codes, each with a fixed, repository-owned message and a fresh stack, never a raw PostgreSQL message/SQLSTATE/SQL/stack/cause/caller value: `projection-runner-unavailable`, `projection-unknown-definition`, `projection-attempt-checkpoint-divergent`, `projection-infrastructure-failed`, `projection-commit-outcome-unknown`. These are distinct from the four stored `safe_error_code` values. The constructor accepts only a code and **normalises it at runtime**, so an `as`-cast secret cannot survive in `.code` any more than in `.message`.

These five codes describe **operational runner failures** once a run is under way. Invalid **invocation input** — a non-canonical injected `now`, or (in the backoff derivation) a malformed/colon-injected projection name or an unsafe/out-of-range version — is a **caller/programming error** and fails closed with the existing repository-wide `ProjectionInputError` **before** a client is acquired (or, for the backoff hash, **before** hashing), rather than being dressed up as one of the operational runner codes. `projectionRetryBackoff` now validates its name through `isProjectionName` and its version through `isProjectionVersion` (a safe integer within range), so the digest is only ever seeded with genuine repository vocabulary; the four exact backoff vectors are unchanged. None of the new modules (runner, lock key, backoff, result, runner errors) is exported from the package root; the event-backbone barrel's API is unchanged.

## Supersession (narrow)

ADR-0019, ADR-0021, ADR-0022, ADR-0034, ADR-0035 and ADR-0036 remain **immutable historical decisions and are not edited.** ADR-0037 supersedes only the following, and nothing else in those decisions:

- **ADR-0021 §2's "Maximum attempts — 5 (configurable)".** The number **five is preserved exactly**; the **"(configurable)"** annotation is superseded. Per ADR-0034 §6 the bound is a **repository constant** (dependency-injectable in tests, never environment-controlled); this slice implements it as such.
- **ADR-0021 §2's Retryable/Terminal handler taxonomy and its `RetryableError` / `TerminalError` handler classes.** For the runner, handler failures are a **single deterministic-handler-failure bucket** (ADR-0034 §3), classified by SQLSTATE **phase** (deterministic handler-side vs infrastructure), not by a handler raising one of two marker error classes. `RetryableError` and `TerminalError` were **design vocabulary only** and are **not implemented in code**; this ADR records that the implemented model is the single-bucket, phase-classified one, with a uniform persisted `projection-handler-failed` attempt code and deterministic persisted jitter.
- **ADR-0034 §7's deferral.** §7 stated the exact `next_attempt_at` interval constants would be **"LOCKED in Stage 3.4.4."** §3 above sets them.
- **ADR-0036's cursor.** The runner's read cursor **and** its reconciliation both key on the projection **position** (`last_position + 1`), never the raw storage `event.sequence`, consistent with ADR-0036.

Everything else in those decisions stands.

## Migration and managed status

- **No new migration.** This is a code-only slice. Migrations `0001`–`0005` are **byte-for-byte unchanged**, and there is **no `0006`**. Stage 3.5's eventual migration still shifts to `0006`+.
- Managed PostgreSQL **remains `0001`-only.** Migrations `0002`, `0003`, `0004` and `0005` are **all unapplied**; because the default migrator applies every pending migration, a managed run would apply `0002 → 0003 → 0004 → 0005`. **No managed migration, preflight, or managed-database access is authorized or performed by this slice.**

## Consequences

**Positive.** ADR-0034 §5's transaction protocol is now executable and proven against real PostgreSQL 17: exactly one position per invocation, an atomic success (attempt + advance in one commit), deterministic persisted retry scheduling, a fifth-failure block, and a fail-closed reconciliation that refuses to treat a divergent checkpoint as progress. The advisory-lock and backoff derivations are deterministic and reproducible from fixed vectors. Hostile handler values cannot leak through the result, force a classification, or poison the pool. The parameters ADR-0034 left open are now locked.

**Negative — accepted.** The runner is a hand-written state machine over a raw client rather than the shared `withTransaction`/`withClient` helpers, because those helpers cannot express "keep the failure bookkeeping, surface the primary error, and destroy a poisoned connection." This is more code, deliberately, at the one layer where a masked error or a leaked connection would be most costly. A single invocation still processes only one event and needs a caller to drive it repeatedly — the worker/scheduler that does so is a later slice.

## Explicit exclusions

Stage 3.4.4 does **not** add: a worker loop, scheduler, or time-driven re-attempt loop; a dead-letter table, quarantine ledger, or replay/unblock/rebuild command (Stage 3.5+); an HTTP endpoint or any `apps/*` change; a live emitter, agent, model, or external integration; a new package-root export; a PostgreSQL enum or any change to migrations `0001`–`0005`; and any Supabase, managed-database, Core, n8n, WhatsApp, or provider access. Managed PostgreSQL still carries only `0001`. The runner is internal vocabulary, verified locally and **pending owner review**; approval of this decision is not authorisation to run it against the managed database, and Stage 3.4.5 is not begun.
